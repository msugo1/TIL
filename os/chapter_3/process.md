### 프로세스 
- `실행 중`인 프로그램
    = active entity
- 현대 컴퓨팅 시스템에서 `작업의 단위`
- 다음에 실행할 명령어를 지정하는 PC 및 관련 자원의 집합을 가진다

### 구조
- 힙, 데이터, 코드, 스택 영역 with PC(Program Counter)

**스택**
- 함수 호출 시 임시데이터 저장장소(함수 매개변수, 복귀 주소, 지역 변수 ...)
- `활성화 레코드`가 스택에 push/pop

**데이터**
- 초기화 된 데이터와 초기화되지 않은 데이터를 저장하는 서로 다른 섹션으로 나뉜다.

NOTE
- main 함수에 전달되는 argc, argv 변수들이 저장되는 별도 섹션이 제공된다.

**Process in Process**
- JVM
    = 다른 개체를 위한 실행환경으로 동작
    = 적재된 코드 해석 -> 코드를 대신하여 원 기계어를 이용하여 행동을 취하는 프로세스

### 상태
NEW - approved -> 준비 <--- interrupt ----------------- 실행---> 종료
                       ---- dispatch (by scheduler) -->
                                    
                  <------------ 대기------------------ (I/O or event)

### PCB (Process Control Block)
- 프로세스 제어블록
- 특정 프로세스와 연관된 정보 수록 
    = 약간의 accounting 데이터와 함께 프로세스를 시작 or 다시 시작시키는데 필요한 모든 데이터를 위한 저장소의 역할

1. 프로세스 상태
- new, ready, running, waiting, halted ..

2. 프로그램 카운터
- 해당 프로세스가 `다음에` 실행할 명령어 주소

3. CPU 레지스터들
- 누산기(accumulator), 인덱스 레지스터, 스택 레지스터, 범용 레지스터 및 상태코드 정보 포함
- PC와 함께 나중에 스케줄링을 위해 인터럽트 발생 시 저장되어야 한다. (올바르게 작업 재개를 위해)

4. CPU 스케줄링 정보
- 프로세스 우선순위, 스케줄 큐에 대한 포인터와 다른 스케줄 매개변수 포함

5. 메모리 관리정보
- base, limit 레지스터 값 & 페이지 테이블 또는 세그먼트 테이블 등 정보 포함
(depending on OS)

6. accounting
- CPU 사용시간, 경과된 실시간, 시간제한, 계정번호, 잡 또는 프로세스 번호 등 포함

7. 입출력 상태정보
- 프로세스에 할당된 입출력 장치들과 열린 파일목록 포함

### Thread
- 프로세스 개념의 확장: one process with multi threads
- 스레드를 지원하는 시스템에서는 PCB가 각 스레드에 관한 정보 포함

### Context Switch
- 스위칭 동안 시스템이 아무것도 못하므로, 컨텍스트 스위칭은 순수한 오버헤드
- 교환 속도는 
1. 메모리의 속도,
2. 반드시 복사되어야 하는 레지스터의 수, 
3. 특수 명령어(ex. 모든 레지스터를 하나의 명령어로 보관하고 적재)
의 존재에 의해서 좌우되므로, 기계마다 다르다.
    = 하드웨어 지원에 크게 좌우된다.
- 컨텍스트 스위칭은 단순히 현행 레지스터 집합에 대한 포인터를 변경하는 것도 포함한다.
    = 레지스터 집합 > 활성 프로세스 인 경우, 시스템은 전처럼 레지스터 자료를 메모리 또는 메모리에서 복사해야 한다.
- 운영체제가 복잡할수록, 스위칭 시 해야할 작업의 양이 더 많아진다.
    ex. 고급 메모리 관리 기법 사용 시, 스위칭 때 더 많은 자료들을 교환해야 한다.

### 부모, 자식 프로세스
- 자식 프로세스 생성 시, 자식 프로세스는 목표 달성을 위한 자원이 필요하다. (CPU 시간, 메모리, 파일, 입출력 장치 ...)
    = 운영체제에게 직접 얻거나 or 부모 프로세스의 자원에 대한 부분집합 활용
    = 부모 프로세스의 자원을 일부만 사용하게 제한 -> 과도한 자식 프로세스 생성으로 인한 과부하 방지
- 물리적, 논리적 자원 제공 외에 부모 프로세스는 자식 프로세스에 초기화 데이터를 전달할 수 있다.
- 자식 프로세스는 부모 프로세스의 `복사본`
    = 똑같은 프로그램 데이터를 가진다.
    = 부모 프로세스가 쉽게 자식 프로세스와 통신할 수 있도록 한다.
    = 자식 프로세스는 열린 파일과 같은 자원 외에 특권과 스케줄링 속성을 부모 프로세스로부터 상속받는다.
- 자식 프로세스가 자신에게 적재될 새로운 프로그램을 가지고 있다.
- `fork()` 를 통해 자식 프로세스 생성
    = 복사본
    = but, fork()의 복귀코드가 다르다. (자식은 부모로 복귀, 새로운 프로세스는 0으로 복귀)
- `fork()` 이후 `exec()` 호출을 통해 자식이 자신의 메모리 공간을 `새로운 프로그램`으로 교체
    = 새 프로그램으로 덮어씌워지므로 exec() 시스템 콜은 오류가 발생하지 않는 한 제어를 반환하지 않는다.
- 부모가 모두 실행된 후에도 자식들이 남아있다면?
    = 준비 큐에서 wait()
    = 자식 프로세스가 모두 끝나면 프로세스 재개 후, exit() 콜로 종료
- 자식 프로세스는 종료 후 `상태` 값을 부모프로세스에 반환
    = 이후 물리 메모리, 가상 메모리, 열린파일, 입출력 버퍼를 포함한 프로세스의 모든 자원이 할당 해제 -> 운영체제로 반납
    = 부모만이 자식을 종료시킬 수 있다. (아니면 오작동 프로세스가 임의로 kill 할 가능성이...)
- 부모가 자식을 종료시키기 위해서는 자식의 pid를 알고 있어야 한다.
    = 자식 생성 시 pid가 부모에게 전달되는 이유

### 자식 프로세스의 종료
1. 자식이 할당된 자원을 초과해서 사용하는 경우
- 부모가 자식들의 상태를 검사할 수 있는 방편이 주어져야 한다.
2. 자식에게 할당된 태스크가 더 이상 필요 없을 때
3. 부모가 exit 하는데, 운영체제는 부모가 exit 한 후에 자식이 실행하는 것을 허용하지 않을 때

### 좀비 프로세스
- 종료되었지만, 부모 프로세스가 아직 wait() 호출을 하지 않은 프로세스
    = 프로세스가 종료하면 사용하던 자원은 운영체제가 되찾아 간다.
    = but, 부모 프로세스가 wait() 호출할 때까지 프로세스 테이블의 해당 항목이 남아있게 된다. 
    = 기본적으로, 종료하게 되면 모든 프로세스는 좀비 프로세스 상태가 된다.
    (물론 아주 짧은 상태로 머무른다.)
    = 부모가 wait() 호출 시, 프로세스 식별자와 프로세스 테이블의 해당 항목이 운영체제에 반환!
- 부모 프로세스가 wait() 호출 대신 종료해버린다면?
    = orphan process!
    = 유닉스에서는 새로운 부모프로세스를 init으로 지정
    = init 프로세스는 주기적으로 wait 호출. orphan process의 종료상태 수집. 프로세스 식별자와 프로세스 테이블의 항목 반환
