### 스레드 스케줄링
- 대부분 최신 운영체제에서는 스케줄 대상이 `커널 수준` 스레드

프로세스-경쟁 범위(process-contention scope)
- 동일한 프로세스에 속한 스레드들 사이에서 CPU 경쟁
    = 스레드가 실제로 CPU 상에서 실행되기 위해서는, 운영체제가 LWP의 커널 스레드를 물리적인 CPU 코어로 스케줄하는 과정이 필요하다.
- PCS는 우선순위에 따라 행해진다.
    = 높은 우선순위를 가진 실행가능한 프로세스 선택

시스템-경쟁 범위(system-contention scope)
- CPU 상에 어느 커널 스레드를 스케줄 할 것인가 결정
- SCS 스케줄링에서의 CPU에 대한 경쟁은 시스템 상의 모든 스레드 사이에서 발생

### 다중 처리기 스케줄링
- 여러 개의 CPU 사용가능? 
    = 부하공유 가능!
    = but, 스케줄링 문제는 더욱 복잡

1. set a master server
- 하나의 처리기가 모든 스케줄링 결정, I/O 처리 및 다른 시스템의 활동 취급 처리
- 다른 처리기들은 사용자 코드만을 수행
(비대칭 다중처리 = asymmetric multiprocessing)

2. symmetric multi-processing, SMP
- 각 프로세서는 스스로 스케줄링
    = 각 프로세스의 스케줄러가 준비 큐 검사 -> 실행할 스레드 선택 -> 스케줄링

    1) 모든 스레드가 공통 준비큐에 있을 수 있다.
    - 공유 준비 큐에 경쟁조건이 생길 수 있다.
    - 두 개의 다른 프로세서가 동일한 스레드를 스케줄링 하지 않도록, 그리고 큐에서 스레드가 없어지지 않도록 보장해야 한다.
    (ex. lock - but 큐에대한 경쟁이 심해져서 병목지점이 될 수 있다.)
    
    2) 각 프로세서는 자신만의 스레드 큐를 가질 수 있다.
    - SMP를 지원하는 시스템에서 가장 일반적인 접근방식
        = 공유 실행큐와 관련된 문제에서 해방
    - 자신만의 프로세스별 큐가 있으면 캐시 메모리를 보다 효율적으로 사용할 수 있다.
    - 큐마다 부하의 양이 다를 수 있다...
        = 균형 알고리즘을 사용해서 모든 프로세서 간에 부하를 균등하게 만들 수 있다.

### 다중 코어 프로세서
- 스케줄링 문제를 더 복잡하게 한다.

memory stall
- 프로세서가 메모리에 접근할 때, 데이터가 가용해지기를 기다리면서 많은 시간을 허비하는 것
    = 프로세서가 메모리보다 훨씬 더 빠르게 동작하므로...
- 캐시 미스로 인해 메모리 스톨이 발생할 수도 있다.
(최대 50% 시간을 허비할 수 있음)

- 위 상황을 해결하기 위해, 최근에는 `다중 스레드 처리코어` 구현
    = 하나의 코어에 2개 이상의 하드웨어 스레드 할당
    = 메모리를 기다리는 동안 하나의 하드웨어에서 스레드가 중단되면, 코어가 다른 스레드로 전환

- 운영체제 관점: 각 하드웨어 스레드는 명령어 포인터 및 레지스터 집합과 같은 구조적 상태 유지
    = 소프트웨어 스레드를 실행할 수 있는 논리적 CPU로 간주
    = 칩 다중 스레딩
    ex. 4개의 코어, 각 코어마다 2개의 스레드 == 8개의 논리적 CPU

### coarse-grained vs fine-grained
1. coarse ...
- 메모리 스톨과 같은 지연시간이 긴 이벤트가 발생할 때까지 한 코어에서 수행
    = 지연시간이 긴 이벤트 발생 시 코어가 다른 스레드 실행
    = 다른 스레드가 수행되기 전, 명령어 파이프라인이 완전히 정리되어야 하므로 스레드간 교환은 비용이 많이듦

2. fine ...
- 명령어 주기의 경계에서 같이 좀 더 세밀한 정밀도를 가진 시점에 스레드 교환 발생
    = 세밀한 시스템의 구조적 설계는 스레드 교환을 위한 회로를 포함
    = 스레드 간 교환 비용 감소


물리적 코어의 자원 (ex. 캐시 및 파이프라인 등)은 하드웨어 스레드 간에 공유되어야 한다.
    = 처리 코어는 한 번에 하나의 하드웨어 스레드만 실행할 수 있다
    = 결과적으로 다중 스레드 다중 코어 프로세스는 현실적으로 두 개의 다른 스케줄링 단계가 필요
    (소프트웨어 스레드 <-> 하드웨어 스레드)

하드웨어 스레드 스케줄링
- 라운드 로빈
- 코어당 2개의 하드웨어 관리 스레드
    = 0 ~ 7까지의 동적 긴급도 배정 (the higher, the more urgent))

두 스케줄링 알고리즘이 함께 사용될 수 있다.
- 운영체제가 프로세서 자원 공유 수준을 알고 있다면
    = 자원을 공유하지 않는 논리 프로세서에 소프트웨어 스레드 스케줄 가능

### 부하 균등화
(= Load Balancing)
- SMP 시스템의 모든 처리기 사이에 부하가 고르게 분배되도록 시도
- 각 처리기가 실행할 스레드를 위한 자기 자신만의 준비 큐를 가진 경우에만 필요
    = 공통 큐에서는 한 처리기가 쉴 때, 다른 처리기가 선택된다.

1. push migration
- 특정 태스크가 주기적으로 각 처리기의 부하 검사
    = 불균형 상태인 경우 과부하인 처리기 -> 덜 바쁜 or 쉬고 있는 처리기로 스레드 이동

2. pull migration
- 쉬고 있는 처리기가 바쁜 처리기를 기다리고 있는 프로세스를 pull

### 프로세스 선호도
- 프로세스 A -> B 이동 시, 캐시 무효화 및 다시 채우는 비용이 많이 든다.
    = 따라서, SMP를 지원하는 대부분의 운영체제는 스레드를 같은 프로세서에서 계속 실행시키면서 `warm cache`를 이용하고 싶어한다.
    = 프로세스 선호도
    = 현재 실행 중인 프로세서에 대한 선호도
- 프로세서 별 준비 큐는 프로세스 선호도를 기본으로 제공
(해당 큐에 담기면 똑같은 프로세서를 사용하므로)

1. 약한 선호도
- OS가 동일한 처리기에서 프로세스를 실행시키려고 노력 (but 반드시 보장X)

2. 강한 선호도
- 시스템 콜을 통해서 프로세스는 자신이 실행될 처리기 집합을 명시할 수 있음

부하 균등이 프로세서 선호도의 이점을 상쇄하기도 한다.
    = 부하 균등을 위해 스레드를 다른 프로세서로 이주시키면서 장점이 사라진다.

### 이기종 다중처리 (Heterogeneous Multiprocessing)
HMP의 목적?
- 작업의 특정 요구에 따라 특정 코어에 작업을 할당하여 전력 소비를 더 잘 관리하는 것
ex. big.LITTLE 
    = 고성능 big 코어가 에너지 효율적인 LITTLE 코어와 결합
    = 백그라운드 작업을 little 코어에 할당해서 더 긴시간 실행가능
