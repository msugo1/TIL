## CPU Scheduling
- 준비 큐에 있는 어느 프로세스에 CPU 코어를 할당할 것인지 결정하는 것

### FCFS (First-Come, Fisrt-Served)
- 비선점형
- simplest
    = 먼저오면 먼저 CPU 할당받음(선착순)
- can be implemented with FIFO
    = 프로세스가 준비 큐에 진입하면, 해당 프로세스의 PCB를 큐의 끝에 연결
    = 이후 CPU가 가용상태가 되었을대, 큐의 맨 앞의 프로세스에 할당
- 실행시간이 긴 프로세스가 앞에 위치하면 대기 시간이 더 길어진다..
    = 응답시간이 중요한 경우 문제가 될 수 있다.
    (대화형 시스템에서는 각 프로세스가 규칙적인 간격으로 CPU를 할당받는 것이 중요)

ex. 24, 3, 3
  24 + (24 + 3) + (24 + 3 + 3)

    vs

    3, 3, 24
  3 + (3 + 3) + (3 + 3 + 24)

### SJF(Shortes-Job-First)
(최단 작업 우선 스케줄링)

- 각 프로세스에 다음 CPU 버스트 길이를 연관시킨다.
    = CPU available -> 버스트가 가장 작은 프로세스에 CPU 할당
- 사실, 다음 CPU의 버스트 시간을 완전히 알기란 불가능...
    = 예측해서 사용하는 것이 보통
    = 측정된 이전 CPU 버스트들의 길이를 지수 평균한 것으로 예측

### RR(Round-Robin)
- 시간 할당량 혹은 타임슬라이스를 가진다.
- 유일하게 실행가능한 프로세스가 아니라면, 연속적으로 두 번 이상의 시간 할당량을 할당받는 프로세스는 없다.
- 성능은 시간할당량의 크기에 많은 영향을 받는다.
    = 시간 할당량 > context switching 시간이어야 한다.
    but 시간 할당량이 너무 커서는 안된다. (그러면 FIFO와 다를게 없다.)

### 우선순위 스케줄링
- 우선순위가 높은 것 먼저! (같으면 FIFO)

**내부적 우선순위**
- 측정가능한 양
ex. 시간 제한, 메모리 요구, 열린 파일의 수, I/O 버스트의 비율 (of CPU burst) ...

**외부적 우선순위***
- 프로세스의 중요성, 컴퓨터 사용을 위해 지불되는 비용의 유형과 양, 작업을 후원하는 부서, 정치적인 요인 등...
- 외부적인 요인

### 다단계 큐 스케줄링
- 프로세스 유형 or 우선순위에 따라 큐를 구별
    = 우선순위가 높은 큐에 있는 것들을 우선적으로 라운드로빈 방식으로 스케줄링
- 각 큐에 자체 알고리즘이 쓰일 수 있다.


## 실시간 CPU 스케줄링
1. 연성 실시간 시스템
- 중요한 실시간 프로세스가 그렇지 않은 프로세스에 대해 우선권을 가지는 것만 보장
(스케줄 되는 시점은 보장X)

2. 경성 실시간 시스템
- 더 엄격한 요구조건!

### 지연시간 최소화
1. 인터럽트 지연시간
- CPU에 인터럽트가 발생한 시점부터, 해당 인터럽트 처리 루틴이 시작하기까지의 시간
    = 인터럽트 발생 -> 우선 수행중인 명령어 완수 -> 인터럽트 종류 결정 -> 인터럽트 서비스 루틴을 사용해서(ISR) 인터럽트 처리 전 현재 수행 중인 프로세스 상태 저장
- 위의 모든 과정이 인터럽트 지연시간
- 인터럽트 지연시간을 줄이는 것이 실시간 운영체제에서 중요
    = 경성(hard) 실시간 시스템에서는 인터럽트 지연시간을 최소화 할 뿐 아니라, 정해진 시간보다 작아야 한다.
    (엄격한 요구조건 충족 위해)
- 커널 데이터 구조체를 갱신하는 동안 인터럽트 불능 = 인터럽트 지연시간에 영향을 주는 요인

2. 디스패치 지연시간
- 스케줄링 디스패처가 하나의 프로세스 블록 -> 다른 프로세스를 시작하는데 까지 걸리는 시간
    = CPU를 즉시 사용하는 실시간 태스크가 있으면 실시간 운영체제는 이 지연시간을 최소화 해야 한다.
- `선점형 커널`: 디스패치 지연시간을 최소화하는 가장 효과적인 방법

**디스패치 지연의 충돌단계**
1. 커널에서 동작하는 프로세스에 대한 선점
2. 높은 우선순위의 프로세스가 필요한 자원을 낮은 우선순위 프로세스 자원이 방출

### 우선순위 기반 스케줄링
- 각각의 프로세스에 중요성에 따라 우선순위 부여
- 스케줄러 선점 기법을 제공하여 현재 CPU를 사용하고 있는 프로세스가 더 높은 우선순위를 갖는 프로세스에 선점될 수 있음
- 선점 및 우선순위 기반의 스케줄러 제공 == 연성 실시간 기능 제공
    = 경성 실시간 시스템에서는 태스크가 마감시간내 확실히 수행되는 것을 보장해야 한다.
    = 부가적인 스케줄링 기법이 필요하다.
    (ex. 승인제어: admission-control = 마감시간 내 완수할 수 있는 프로세스만 실행허용)

### Rate-Monotonic 스케줄링
- 선점 가능한 정적 우선순위 정책 이용. 주기 태스크들 스케줄
    = 낮은 우선순위 프로세스 ~ing, 높은 우선순위 프로세스 now ready -> 높은 우선순위 프로세스 선점
- 주기가 짧으면 높은 우선순위, 길면 낮은 우선순위
    = CPU를 더 자주 필요로 하는 태스크에 더 높은 우선순위 부여

### Earliest-Deadline-First 스케줄링
- EDF
    = 마감시간에 따라서 우선순위를 동적으로 부여
    (마감시간이 빠를수록 높은 우선순위)
- 프로세스가 실행 가능하게 됨 -> 자신의 마감시간을 시스템에 알림
    = 우선순위가 새로 실행하게 된 프로세스의 마감시간에 맞춰서 다시 조정
- 프로세스들이 주기적일 필요 X. CPU 할당 시간도 상수 값으로 정해질 필요 X.
    = 이론적으로는 최적
    (이론적으로는 모든 프로세스가 마감시간을 만족시키도록 할 수 있고, CPU 이용률 역시 100%가 될 수 있음)    
    = 실제로는 프로세스 사이 or 인터럽트 핸들링 시 문맥교환 때문에 100% 이용 불가능

### 일정 비율의 몫 스케줄링
(proportional share)

- 모든 응용들에 T개의 시간 몫 할당
ex. A: 50, B: 15, C:20
    = 처리기 시간 중 각각의 몫(퍼센트)만큼 가져간다.

- 응용이 시간 몫을 할당받는 것을 보장하는 승인 제어정책과 함께 동작해야 한다.

**승인 제어정책**
- 사용 가능한 충분한 몫이 존재.
- 그 범위 내의 몫을 요구하는 클라이언트들에게만 실행 허락.
ex. A + B + C = 85
    D: 30 이 몫을 요구하면 거부

### POSIX 실시간 스케줄링
1. SCHED FIFO
- FIFO
    = 큐 가장 앞의 실시간 스레드가 가장 높은 우선순위를 가진다.
    = 해당 스레드가 종료되거나 블록 될 때까지 CPU 할당

2. SCHED RR
- Round Robin
    = 같은 우선순위의 스레드에 시간 할당량 제공
    = 나머지는 SCHED_FIFO와 비슷

3. SCHED OTHER
- 구현이 정해지지 않아 시스템마다 다름

### CFS
= Completely Fair Scheduler
(완전 공평 스케줄러)

- 처리 코어간의 부하를 균등하게 유지 + NUMA 인식 + 스레드 이주 최소화
- 스레드의 우선순위 + 평균 CPU 이용률의 조합으로 각 스레드 부하 정의

스레드 이주 시 불이익 == 캐시 무효화
- NUMA 시스템에서 메모리 액세스 시간이 더 길어진다.
    = 메모리 액세스 불이익
- Linux는 이 문제를 해결하기 위해 스케줄링 도메인의 `계층적 시스템`을 결정
    = CFS는 가장 낮은 수준의 층부터 시작해서, 도메인 안에서 부하의 균형을 맞춘다.
