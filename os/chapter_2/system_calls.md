### 시스템 콜
- OS에 의해 사용가능하게 된 서비스에 대한 인터페이스 제공
ex. 하드웨어 직접 접근
(written in Assembly or C or C++)

ex. cp file1 file2
- 다양한 시스템 콜이 일어난다.
(파일 오픈부터, 오류 발생시 처리, 파일 복사완료 후 닫기 및 콘솔에 출력 ...)

- 각 운영체제 시스템 콜마다 필요한 정보의 유형과 양이 상이하다.

### API (Application Programming Interface: 응용 프로그래밍 인터페이스)
- 간단한 프로그램이라도 운영체제의 기능을 아주 많이 사용하게 된다.
ex. 초당 수천 개의 시스템 콜 수행
- 대부분 응용 개발자들은 `응용 프로그래밍 인터페이스`에 따라 프로그램 설계
    = 각 함수에 전달되어야 할 `매개변수`, 프로그래머가 기대할 `반환 값`을 포함 ...
- 뒷단에서 API를 구성하는 함수들이, 응용 프로그래머를 `대신`하여 `실제 시스템 콜 호출`

### Why API Call > Direct System Call?
- 호환성
    = API에 맞게 설계되면, 같은 API를 지원하는 대부분 시스템에서건 컴파일/실행가능
- 시스템 콜은 종종 더 자세한 명세가 필요
- 프로그램 상에서 작업하기가 응용 프로그래머에게 가용한 API보다 더 어렵다.

### RTE(실행시간 환경)
- 운영체제가 제공하는 시스템 콜에 대한 연결고리 역할을 하는 `시스템 콜 인터페이스` 제공
    = 시스템 콜 인터페이스는 API 함수의 호출을 가로채 필요한 운영체제 시스템 콜을 부른다.
    = 각 시스템 콜에는 `번호`가 할당된다.
    = 시스템 콜 인터페이스는 이 번호에 따라 `색인`되는 테이블을 유지한다.
- 시스템 콜 인터페이스는 의도하는 시스템 콜을 부르고, 시스템 콜의 상태와 반환 값을 돌려준다.
- 호출자는 API를 준수하고, 시스템 콜의 결과로서 운영체제가 무엇을 할 것인지만 이해하면 된다.
    = 운영체제 인터페이스에 대한 자세한 내용은 API에 의해 프로그래머로부터 숨겨진다.
    = 이것은 다시 RTE에 의해 관리된다.

### 운영체제에 매개변수 전달
1. 매개변수를 레지스터 내에 전달
2. (레지스터 수보다 더 많은 매개변수가 있는 경우) 메모리 내의 블록 or 테이블에 저장
    = 블록의 `주소`가 레지스터 내 매개변수로 전달된다.

매개변수는 프로그램에 의해 스택에 push, 운영체제에 의해 pop

## 시스템 콜의 유형(p. 73)
: 프로세스 제어, 파일 조작, 장치 조작, 정보 유지, 통신/보호

### 1. 프로세스 제어
- 실행중인 프로그램은 수행을 정상적으로 혹은 비정상적으로 멈출 수 있어야 한다. (end() or abort())
- 적재된 프로그램이 종료되었을 때?
    1) 기존 프로그램으로 제어 back
        = 반드시 기존 프로그램의 메모리 이미지를 보관해야 한다.
        = 한 프로그램이 다른 프로그램을 호출하는 기법을 만든 셈
    2) 두 프로그램 병행 수행 
        = 다중 프로그래밍 될 새로운 프로세스 생성한 것
    ...
- 새로운 잡이나 프로세스, 혹은 잡들이나 프로세스들의 집합을 생성한다면?
    = 실행을 제어할 수 있어야 한다.
    by 우선순위, 최대 허용 실행시간 ...
    = 잡 혹은 프로세스의 속성들을 결정하고 재설정할 수 있는 능력이 필요
    = 잘못되었거나 더 이상 필요 없는 경우 종료할 수 있어야 한다.
- 새로운 프로세스 생성 후에는 실행이 끝나기를 기다려야할 필요가 있을 수 있다.
    = 일정 시간만큼(wait_time()) or 특정 이벤트가 일어날 때까지(wait_event()) invoked by signal_event()

- 둘 이상의 프로세스들은 데이터를 공유한다.
    = 공유되는 데이터의 `일관성`을 보장하기 위해, 운영체제는 종종 프로세스가 공유데이터를 `잠금`할 수 있는 시스템 콜을 제공한다.

### 2. 파일관리
- 파일을 생성, 삭제할 수 있어야 한다. (create, delete)
    = 파일 이름 or 파일 속성의 일부 요구
    (ex. 유형, 보호코드, 회계정보..)
- 파일 생성 후에는 열고 사용 (open)
    = 읽고, 쓰고, 위치변경 ... (read, write, reposition, rewind)
    = move or copy ..
- 파일 닫기 (close)
- 파일 시스템이 파일을 조직하기 위해 디렉터리 구조를 가진다?
   = 디렉터리에 대해서도 위와 같은 연산집합이 필요하다.
- 파일이나 디렉터리에 대해 여러 속성의 값을 결정할 수 있어야 하고, 필요하다면 재설정도 가능해야 한다.

### 3. 장치관리
- 프로세스는 작업을 계속 수행하기 위해 추가 자원이 필요할 수 있다.
    = 이러한 자원은 주 기억장치, 디스크 드라이브, 파일에의 접근 등이 될 수 있다.
- 운영체제에 의해 제어되는 다양한 자원들은 `장치`로 간주될 수 있다.
    = 물리장치, 가상장치 ...
    = 다수의 사용자가 동시에 사용하는 시스템은 독점적인 장치 사용을 보장받기 위해 장치를 요청하기를 요구한다. (request!)
    = 장치의 사용이 끝나면 반드시 release
    (파일 open, close와 비슷)
- 입출력 장치와 파일 간에는 유사성이 많다.
    = 따라서 많은 운영체제에서 이들을 파일 - 장치 구조로 본다.
    = 같은 시스템 콜이 파일과 장치에 대해 사용된다.

### 4. 정보유지 관리
- 많은 시스템콜: 단순히 사용자 프로그램과 운영체제 간의 정보 전달을 위해 존재
- 다른 시스템 콜: 운영체제의 버전 번호, free memory, free disk 공간 등과 같은 시스템에 대한 정보 제공
    = memory dump 시스템 콜 제공 (디버깅 시 유용)
- 또한, 많은 운영체제가 프로그램의 `시간 프로파일` 제공
    = 프로그램이 특정 위치 혹은 위치의 집합에서 수행한 시간의 양을 나타낸다.
    = 시간 프로파일은 추적 설비 or 정규 타이머 인터럽트가 필요하다.
    = 타이머 인터럽트가 발생할 때마다 `프로그램 카운터` 값이 기록된다.
    = 타이머 인터럽트가 충분히 빈번하게 발생한 경우, 프로그램의 여러 부분에서 소비한 시간의 통계적 그림을 얻을 수 있다.
- 운영체제는 현재 운영되고 있는 모든 프로세스에 관한 정보를 가지고 있다.
    = 이러한 정보에 접근하기 위한 시스템 콜도 있다.

### 5. 통신
1) 메시지 전달
- 통신하는 두 프로세스가 정보를 교환하기 위하여 서로 메시지를 주고 받는다.
    = 메시지는 두 프로세스 사이에 직접 교환되거나 우편함을 통하여 간접적으로 교환될 수 있다.
    = 각 프로세스도 컴퓨터의 호스트 이름과 마찬가지로 `프로세스 이름`을 지닌다.
    = 운영체제에 의해 해당 이름은 `동등한 식별자`로 반환된다.
    = 반환된 식별자는 운영체제가 프로세스를 가리키는데 사용할 수 있다.
- 소량의 데이터 교환 시 유용
    = 피해야 할 충돌이 없으므로
- 메모리 공유보다 구현하기 쉽다.

2) 공유 메모리
- 다른 프로세스가 소유한 메모리 영역에 접근하기 위해 `shared_memory_create()`, `shared_memory_attach()` 시스템 콜 사용!
    = 일반적으로 운영체제는 한 프로세스가 다른 프로세스의 메모리에 접근하는 것을 막는다.
    = 공유 메모리는 두 개 이상의 프로세스가 이러한 제약을 없애는데 동의해야 한다.
- 제약이 없어진 프로세스들은 공유 영역에 데이터를 읽고 씀으로써 정보를 교환할 수 있다.
    = 데이터 형식은 프로세스에 의해 결정 (not under os's control)
- 프로세스는 동일한 위치에 동시에 쓰지 않도록 보장할 책임을 진다.
- 한 컴퓨터 안에서는 메모리 전송 속도로 수행이 가능하다.
    = 최대 속도와 편리한 통신 허용
    = but, 보호 및 동기화 부분에서 여러 문제점을 지닌다.

### 6. 보호
- 컴퓨터 시스템이 제공하는 자원에 대한 접근 제어
- 네트워킹/인터넷의 출현으로 서버에서 휴대용 컴퓨터까지 모든 컴퓨터 시스템에서 보호를 고려해야 한다.
ex. set_permission(), get_permission()
    = 파일과 디스크와 같은 자원의 허가 권한 설정 시 이용

### Loader
- 이진 실행 파일을 메모리에 적재하는데 사용
    = CPU 코어에서 실행할 수 있는 상태가 된다.
- 재배치?
    = 프로그램 부분에 최종주소 할당. 프로그램 코드와 데이터를 해당 주소와 일치하도록 조정.
    = 프로그램이 실행될 때 코드가 라이브러리 함수를 호출하고, 변수에 접근할 수 있게 한다.

ex. ./main.out
fork(): 프로세스 생성
    -> exec(): 로더 호출, 및 실행파일 이름 전달
    -> 로더가 새로 생성된 프로세스의 주소공간 사용, 지정된 프로그램 메모리에 적재

### ABI (application binary interface)
- 이진코드의 여러 구성요소가 주어진 아키텍처에서 특정 운영체제와 상호작용할 수 있는 방법 정의
    = 주소길이, 시스템 콜에 매개변수 전달법, 런타임 스택 구성, 시스템 라이브러리의 이진 형식 및 데이터 유형의 크기 등 ...
    = 하위 수준의 세부정보 명시!
- 특정 아키텍처에 의해 명시된다.
    = 아키텍처 수준의 API
    = 특정 아키텍처에서 실행되는 특정 운영체제에 대해 정의되어 있으므로 no cross platform

### 부팅과정(Linux)
전원 -> BIOS/UEFI 위치 감지
     -> 장치들 검사(CPU, Memory ... 이상 없는지)
     -> Linux Kernel 이미지 to Memory (압축 되어있다고 한다.)
     -> 커널 이미지 압축해제, 임시 RAM 파일 시스템 생성(initramfs)
     -> 커널 시작 후 필요한 드라이버 설치
     -> 루트 파일 시스템 이동(initramfs to 적절한 루트파일 시스템 위치)
     -> systemd 프로세스 생성 및 다른 서비스 시작
     -> 사용자에게 로그인 프롬프트 표시

### 성능 관찰 및 조정
1. 카운터
- 카운터를 통해 호출된 시스템 콜 횟수 또는 네트워크 장치 또는 디스크에 수행된 작업 수와 같은 시스템 활동 추적
ex. ps, top(process), vmstat(memory), netstat(network), iostat(disk io)
- /proc 파일 시스템에서 통계를 읽어온다.

**/proc**
- 커널 메모리에만 존재하는 `의사` 파일시스템 (pseudo file system)
    = 다양한 프로세스별 통계와 커널 통계 질의시 사용
- /proc 파일 시스템은 디렉터리 계층구조로 구성
    = 프로세스가 /proc 하위 디렉터리로 표시
    ex. 2155번 프로세스 (/proc/2155)

2. 추적
- 시스템 콜과 관련된 단계와 같은 `특정 이벤트`에 대한 데이터 수집

ex.
프로세스별: strace(프로세스에 의해 호출된 시스템 콜 추적), gdb(소스 레벨 디버거)
시스템 전체: perf(리눅스 성능 도구 모음), tcpdump(네트워크 패킷 수집)
