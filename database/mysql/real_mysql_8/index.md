## 데이터 저장방식으로 구분
### 1. B-Tree
- 루트노드 -> 자식노드 구조(자식노드 consists of 브랜치 노드 & 리프노드)
- 인덱스의 리프노드는 항상 실제 데이터 레코드를 찾아가기 위한 `주소값`을 가진다.
    = 인덱스와 실제 데이터는 따로 관리
    = InnoDB에서는 `PK`가 ROWID 역할을 한다.

**인덱스 키 추가 in B-tree**
1. 저장될 키 값을 이용해 적절한 위치 탐색
- 위치 결정시 `레코드 키 값 + 대상 레코드 주소` in a leaf node
2. 리프노드가 꽉찼으면 분리
- 상위 브랜치까지 전파

**인덱스 키 삭제 in B-tree**
- 키 값이 저장된 리프노드 검색 후 삭제

**인덱스 키 변경 in B-tree**
- 키 값에 따라 저장될 위치 달라짐
    = 단순 변경은 불가하고, only 삭제 -> 추가 형식의 작업

**인덱스 키 검색 in B-tree**
- DML 시 추가 비용에도 B-Tree 인덱스 쓰는 이유
- 인덱스 검색이 유효한 경우
    1) 100% 일치
    2) 앞 부분일치

**index range scan**
1. index seek
- 조건을 만족하는 값이 저장된 위치 찾기
2. index scan
- 탐색된 위치부터 필요한 만큼 인덱스를 차례대로 읽기
3. 실제 레코드 읽어오기
- 1~2에서 얻은 인덱스 키, 레코드 주소를 바탕으로 레코드가 저장된 페이지를 가져와 최종 레코드를 읽어온다.

then **covering index**
- 위의 과정에서 3번은 각 건마다 random i/o
    = 요거를 줄일 수 있다면?
- 인덱스를 잘 설정해둬서, 1 ~ 2번 만으로도 필요한 데이터를 다 가져올 수 있다면
    = 3번은 불필요
    = 추가 랜덤io 불필요
    = 성능향상!

**index full scan**
- 인덱스의 처음부터 끝까지 모두 읽어야 하는 경우
- 인덱스크기가 테이블 크기보다 작으므로 테이블 풀 스캔보다 효율적
- 쿼리가 인덱스에 명시된 컬럼만으로 처리가능할 때
    = 더 적은 I/O
- still, 인덱스의 비효율적인 사용

**loose index scan**
- 듬성듬성 필요한 인덱스만 read
    = 중간에 필요치 않은 키 값은 스킵

**index skip scan?**
- upgrade 된 loose index scan since 8.0

example
```
ALTER TABLE employees
    ADD INDEX ix_gender_birthdate (gender, birth_date)

SELECT * FROM employees WHERE gender = 'M' AND birth_date >= '1965-02-01';
(인덱스 사용 가능)

SELECT * FROM employees WHERE birth_date >= '1965-02-01';
(인덱스 only possibe from 8.0 with SET optimizer_switch='skip_scan=on')
```
- 인덱스에 존재하는 모든 값 추출 -> 결과를 이용해 인덱스 스킵 스캔
    (8.0 현재 limit)
    = WHERE 절에 명시되지 않은 인덱스 컬럼에 대한 유니크 값이 적어야 한다...
    = 쿼리가 인덱스에 존재하는 컬럼만으로 처리가능해야 한다. (SELECT 조회 대상이 인덱스에 모두 있는 값들이어야 한다.)

**multi-column index**
- 2개 이상의 컬럼으로 구성된 인덱스(= `concatenated index`)
- 인덱스의 두 번째 컬럼은 첫 번째 컬럼에 `의존해서 정렬`
    = 두 번째 컬럼의 정렬은, 첫 번째 컬럼이 똑같은 레코드에서만 의미가 있다.
    = 인덱스 내 컬럼의 `순서`가 중요한 이유!

### 2. R-Tree 인덱스
- `Spatial Index`
    = 2차원 데이터 인덱싱 & 검색
- 공간확장을 이용해 간단하게 위치기반 서비스 구현가능?
    = 공간확장 includes the following three: 공간 데이터를 저장할 수 있는 데이터 타입, 공간 데이터의 검색을 위한 공간 인덱스, 공간 데이터의 연산함수    

1) 도형 정보를 관리할 수 있는 데이터 타입
- point, line, polygon, geometry
    = geometry is a super type of all
- MBR(Minimum Bounding Rectangle)
    = 해당 도형을 감싸는 최소 크기의 사각형
    = 이 사각형들의 포함관계를 B-Tree 형태로 구현한 인덱스 = R-Tree 인덱스'

2) R-Tree 인덱스 용도
- 공간 인덱스
    = 일반적으로는 WGS84(GPS) 기준의 위도, 경도 좌표 저장에 주로 사용
- 각 도형의 포함관계를 이용해 만들어진 인덱스
    = ST_Contains(), ST_Within() 등과 같이, `포함관계`를 비교하는 함수로 검색을 수행하는 경우에만 인덱스 사용가능
    ex. 현재 사용자로부터 5km 반경 이내의 음식점 검색

### 3. 전문 검색 인덱스(p. 258)
- 문서의 내용 전체 인덱스화 -> 특정 키워드가 포함된 문서 검색
    = 일반적인 B-Tree 인덱스 사용불가

**인덱스 알고리즘**
1) 어근 분석 알고리즘
- 어근 분석: 검색어로 선정된 단어의 뿌리인 원형을 찾는 작업
    = 어근 분석보다는 형태소 분석 후 명사/조사를 구분하는 기능이 더 중요하다.
    = 각 국가의 언어, 문법이 다르므로 형태소 분석 or 어근 분석 또한 언어별로 방식이 다르다.
    = 플러그인 적용은 어렵지 않지만, 완성도를 갖추는 것이 힘들다.

2) n-gram 알고리즘
- 단순히 키워드를 검색해내기 위한 인덱싱 알고리즘
    = 본문을 무조건 몇 글자씩 잘라서 인덱싱
    = simple but bigger sized index
- 일반적으로 `2`글자 단위로 쪼개서 인덱싱(= 2-gram)

**전문 인덱스를 사용하려면?**
1) 쿼리 문장이 전문 검색을 위한 문법 사용(ex. MATCH ... AGAINST ...)
2) 테이블이 전문검색 대상 컬럼에 대한 전문 인덱스 보유

### 4. 함수 기반 인덱스
(available since 8.0)
- 컬럼의 값을 변형해서 만들어진 값에 대한 인덱스

1) 가상 칼럼을 이용한 인덱스(p. 266)
ex.
```
CREATE TABLE user (
    user_id BIGINT,
    first_name VARCHAR(10),
    last_name VARCHAR(10),
    PRIMARY KEY (user_id)
);

to create an index for full name(first name + last name)
- before 8.0, full_name 칼럼 추가 후 실제 값을 추가해야 인덱싱가능
- since 8.0, though, 가상 칼럼 추가 후 가상칼럼 인덱스 생성가능

ALTER TABLE user
    ADD full_name VARCHAR(30) AS (CONCAT(first_name,' ',last_name)) VIRTUAL,
    ADD INDEX ix_fullname (full_name);
```
**주의**
- 가상 칼럼은 테이블에 새로운 컬럼을 추가하는 것과 같은 효과
    = 실제 테이블의 구조 변경

2) 함수를 이용한 인덱스
- since 8.0, 테이블 구조 변경 없이도 함수를 직접 사용하는 인덱스 생성가능

- 인덱싱할 값을 계산하는 과정에만 차이
    = 내부 구조 및 유지관리 방법은 B-Tree 인덱스와 동일

### 5. 멀티밸류 인덱스
- 전문 검색 인덱스를 제외한 모든 인덱스
    = 레코드 1건이 1개의 인덱스 키 값을 가짐
    = 인덱스 키와 데이터 레코드의 관계 = 1 : 1
- 멀티 밸류 인덱스
    = 하나의 데이터 레코드가 여러 개의 키 값을 가질 수 있는 형태
    = 정규화에 위반 but JSON 지원을 위해
    (JSON 배열 타입의 필드에 저장된 원소들에 대한 인덱스)
- 반드시 다음의 함수들을 이용해서 검색해야 한다.
1) MEMBER_OF()
2) JSON_CONTAINS()
3) JSON_OVERLAPS()

### 6. 클러스터링 인덱스
**클러스터링**
- 여러 개를 하나로 묶음
    = 테이블의 레코드를 비슷한 것끼리 묶어서 저장(based on Primary Key)

1) 클러스터링 인덱스
- 클러스터링 인덱스는 테이블의 PK에 대해서만 적용된다.
    = PK 값이 비슷한 레코드끼리 묶어서 저장하는 것
- PK 값에 의해 레코드의 저장위치가 결정된다.
    = PK가 변경되면, 레코드의 물리적인 저장위치가 바뀌어야 한다.
    = PK 의존도가 높으므로, 신중히 PK 결정
- InnoDB는 항상 클러스터링 인덱스로 저장
    = PK 기반의 검색이 매우 빠르다.
    = 레코드의 저장 or PK 변경이 상대적으로 느리다.

**PK가 Secondary Index에 미치는 영향**
**InnoDB 테이블 기준, 세컨더리 인덱스가 실제 레코드가 저장된 주소를 가진다면?**
- 클러스터링 키 값이 변경될 때마다, 데이터 레코드의 주소 변경
- 그때마다 해당 테이블의 모든 인덱스에 저장된 주솟값 변경필요
    = 오버헤드
- 따라서, 모든 세컨더리 인덱스는 해당 레코드가 저장된 주소가 아니라, PK 값을 저장

**클러스터링 인덱스 장단점**
1) 장점
- PK검색 처리성능
- 커버링 인덱스(세컨더리 인덱스가 PK를 가지고 있어, 인덱스 만으로 처리될 수 있는 경우가 많다.)

2) 단점
- 모든 세컨더리 인덱스가 클러스터링 키를 갖는다.
    = 클러스터링 키 값이 클 경우, 인덱스 크기가 전체적으로 증가
- 세컨더리 인덱스 -> PK -> 실제 레코드
    = 약간의 처리성능 저하
- INSERT 시, PK에 의해 레코드 저장위치가 결정되므로 처리성능 저하
- PK 변경시 레코드를 DELETE -> INSERT 해야 하므로, 처리성능 저하

**클러스터링 인덱스 주의사항**
1) 클러스터링 인덱스 크기
- 클러스터링 인덱스 크기 증가 - 세컨더리 인덱스 크기 증가
    = 저장공간/버퍼의 비효율적인 사용
  
  2) PK는 반드시 명시
  - 어차피 내부적으로 PK 생성된다. (이 경우는 컨트롤 불가 == PK로 얻을 수 있는 이득 0)

### 7. 유니크 인덱스
- 인덱스 보다는 제약조건에 더 가깝다.
    = 테이블이나 인덱스에 같은 값이 2개 이상 저장될 수 없음
    (주의: NULL은 값이 아니므로 2개 이상 가능)

**유니크 인덱스 vs 세컨더리 인덱스**
1. 읽기
- 유니크 인덱스가 빠르지만은 않다.
    = 1건만 읽어도 되서 더 빠르다는 말은 ㄴㄴ
- 읽어야 할 레코드 건수가 같다면 성능차이X

2. 인덱스 쓰기
- UK는 중복값 체크과정이 필요
    = 쓰기 시 더 느리다.
    = 체인지 버퍼 사용불가
- 유니크 인덱스에서 중복된 값을 체크할 때는 S락, 쓰기 시에는 X락
    = 데드락이 자주 발생

**유니크 인덱스 사용시 주의사항**
1. 불필요한 유니크 인덱스는 생성X
    = 꼭 유니크함이 보장되어야 하는 경우만
    = 성능이 더 좋아지지 않는다.
2. 유니크 인덱스, 세컨더리 인덱스를 중복생성 X
    = 둘 다 같은 역할
3. PK와 동일한 유니크 인덱스 또한 불필요한 중복

### 8. 외래키
- 제약조건
1) 테이블의 변경이 발생하는 경우에만 잠금 경합 발생(X Lock)
2) 외래키와 연관되지 않은 칼럼의 변경은 최대한 X Lock X

- 외래키 설정 시, 자동으로 연관되는 테이블의 칼럼에 인덱스 생성
- 외래키가 제거되지 않은 상태에서 자동으로 생성된 인덱스 삭제 불가
- 물리적인 외래키 생성은 체크를 위한 연관 테이블에 `추가잠금`을 걸게 되므로, 잠금 경합까지 고려해 모델링을 진행해야 한다

### 제대로 된 인덱스 설정이 중요한 이유?
- InnoDB는 검색을 수행한 인덱스 잠금 -> 테이블 레코드 잠금
- 인덱스가 제대로 설정되어있지 않다면, 불필요한 레코드 or even 전체 레코드 잠금

### 인덱스 크기?
- 인덱스 크기가 커지면,
    
    1) 한 페이지(버퍼링 or 저장 기본단위)에 들어갈 수 있는 인덱스 크기가 줄어든다.
        = 더 많은 디스크 IO
    2) B-Tree 깊이가 깊어진다.
        = 하나의 페이지에 담을 수 있는 개수가 줄어듬 -> 깊이가 깊어짐 -> 더 많은 탐색

### Cardinality
- 모든 인덱스 키 값 가운데, 유니크한 값의 수
- 인덱스 키 값 가운데 중복된 값이 많을 수록 기수성이 낮아지고, 선택도도 떨어진다.
- 선택도가 높을수록 검색 대상이 줄어드므로, 그만큼 빠르게 처리된다.
- 다만, 선택도 말고도 `정렬` or `그루핑`을 위해 인덱스를 만드는 것이 나은 경우도 있으니 case by case
- 인덱스에서 `유니크한 값`의 개수는 인덱스의 쿼리나 효율성에 큰 영향을 미친다.


### 인덱스 정렬
- 8.0부터는 오름차순/내림차순 혼합 정렬 된 멀티 컬럼 인덱스 생성이 가능
    = 인덱스 각각 컬럼이 혼합된 경우도 문제 없이 정렬가능

**though 인덱스 역순 스캔이 더 느리다.**
Why?
1. 페이지 잠금이 인덱스 정순 스캔에 적합
2. 페이지 내에서 인덱스 레코드가 `단방향`으로만 연결되어 있다.

### B-Tree 인덱스의 가용성과 효율성
- 작업 범위 결정조건 vs 필터링/체크 조건
- 전자를 잘 맞춰줘야 작업 범위를 줄여줄 수 있다.

1. 인덱스의 가용성
- B-Tree 인덱스 
    = 왼쪽 값에 기준해서 오른쪽 값 정렬
    = 왼쪽 칼럼의 값을 모르면 인덱스 레인지 스캔 불가!

2. 효율성 판단(작업범위 결정조건을 사용할 수 없는 경우)
- NOT-EQUAL 비교
    = <>, NOT IN, NOT BETWEEN, IS NOT NULL

- LIKE %??
    = 뒷 부분일치 형태

- 인덱스 컬럼 변형 후 비교

- NOT-DETERMINISTIC 속성의 스토어드 함수가 비교조건으로

- 데이터 타입이 서로 다른 비교
    = 인덱스 컬럼 타입을 변환해야 비교가 가능한 경우

- 문자열 데이터 타입의 콜레이션이 다른경우
    ex. utf8 vs euckr
