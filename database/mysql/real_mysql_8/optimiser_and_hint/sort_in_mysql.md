## 기본 데이터 처리
### 풀 테이블 스캔 & 풀 인덱스 스캔
1. 풀 테이블 스캔을 선택하는 경우?
- 테이블의 레코드 건수가 적어서, 풀 테이블 스캔이 빠른 경우(테이블 = 페이지 1개)
    = 테이블의 전체크기 > 인덱스 크기 in general
    = therefore, 풀 테이블 스캔 실행 시 한꺼번에 여러 블록 or page를 읽어오는 기능이 있다.
    = `read ahead`
    = 어떤 영역의 데이터가 앞으로 필요해지리라는 것을 예측 -> 요청이 오기 전에 미리 디스크에서 읽어 버퍼풀에 장착
    = 풀 테이블 스캔 시, 처음 몇 페이지는 read by foreground thread but at one point, it is the background thread that takes charge
    = 4 ~ 8 개부터 점차 읽어오는 페이지 수를 증가 시킨다. max - 64
    = 미리 읽어와 풀에 저장하기 때문에, 어느 순간 부터 쿼리처리가 빨라진다.
    = `innodb_read_ahead_threshold`: 언제 리드 어헤드를 시작할지
- WHERE or ON 절에 인덱스를 활용할 수 있는 적절한 조건이 없는 경우
- range를 활용할 수 있더라도, 옵티마이저가 판단한 조건 일치 `레코드 건수가 너무 많은 경우`

2. 풀 인덱스 스캔
- 인덱스를 처음부터 끝까지 스캔

## ORDER BY(Using Index or Filesort)
### 정렬 시 인덱스 사용이 불가능한 경우?
- 정렬 기준이 너무 많아서 요건 별로 모두 인덱스 생성이 불가능한 경우
- GROUP BY의 결과 또는 DISTINCT 같은 처리 결과를 정렬해야 하는 경우
- UNION 결과와 같이 임시 테이블의 결과를 다시 정렬해야 하는 경우
- 랜덤하게 결과 레코드를 가져와야 하는 경우

### index vs filesort
1. index
장점
- 정렬되어 있으므로 위 혹은 아래부터 읽기만 하면 끝!

단점
- DML 시 부가작업 -> 추가 시간 and 비용
- 인덱스 때문에 디스크 공간이 더 필요
- 인덱스 개수가 늘어날 수록 버퍼 풀을 위한 메모리 공간도 더 필요

2. filesort
장점
- 인덱스 생성 시 단점X
- 정렬할 레코드가 많지 않으면, 메모리에서 Filesort 처리되므로 충분히빠르다.

단점
- 레코드 대상 건수가 많아질 수록 쿼리 응답속도 저하
    = why? 정렬 작업이 쿼리 실행시 처리되므로

### sort buffer
- 정렬을 수행하기 위한 별도의 메모리 공간
- 정렬이 필요한 경우만 할당된다.
    = 쿼리 실행이 완료되면 해당 메모리 공간 반납!
- 최대 사이즈는 `sort_buffer_size` 변수에 의해 결정된다.

### 정렬이 문제가 되는 경우?
- 정렬해야 할 레코드 건수 크기 > 할당된 소트 버퍼 크기
- 레코드를 여러 조각으로 나누어서 처리해야 한다.
- But, 이때 임시저장을 위해 `디스크`를 사용한다.
    = 레코드 조각 정렬 in memory -> 결과는 디스크에 임시저장 repeatedly
    = 이후 각 버퍼 크기만큼 정렬된 레코드를 병합하면서 다시 정렬 수행
    = 디스크 IO 유발
    = 레코드 건수가 많을수록 IO 증가
- 결국 디스크를 사용하는 것이 문제!
- `256KB < size < 8MB` 경우 최고 성능
    = 너무 큰 버퍼를 사용하면 메모리 공간 할당이 커져 성능저하 발생가능성
    = and, 소트 버퍼는 세션 메모리 영역에 해당한다. 따라서, 각 클라이언트가 별도의 공간을 가진다.
    (크기가 커질수록 각각 할당되는 메모리 크기가 커져서 OOM 위험)

### 정렬 알고리즘
1. <sort_key, rowid>: 정렬 키와 레코드의 로우 아이디만 가져와서 정렬 = two-pass
2. <sort_key, additional_fields>: 정렬 키와 레코드 전체를 가져와서 정렬, 레코드 컬럼들은 고정 사이즈로 메모리 저장 = single-pass
3. <sort_key, packed_additional_fields>: 정렬 키와 레코드 전체를 가져와서 정렬, 레코드의 컬럼들은 가변 사이즈로 메모리 저장 = single-pass

1. single-pass
- SELECT 대상이 되는 컬럼 전부를 담아서 정렬 수행

2. two-pass
- 정렬대상 컬럼과 PK 값만 소트버퍼에 담아서 정렬수행
- 정렬된 순서대로 다시 PK를 사용해 테이블을 읽어서 SELECT 할 컬럼을 가져온다.

두 번 읽어야 하거나(two-pass), 더 많은 소트 버퍼 공간을 필요로 하거나(single-pass)
- 최신 버전에서는 일반적으로 single-pass
- two-pass를 사용해야 하는 경우?
1) 레코드 크기 > max_length_for_sort_data
2) BLOB or TEXT Column included

**결론**
- 꼭 필요한 컬럼만 가져와 쓰자
- 다 가져오면 정렬 버퍼만이 아니라, 임시 테이블이 필요한 쿼리에서도 영향을 미친다 ㅜ

### 정렬 처리방법
1. 인덱스를 사용한 정렬
```
1) ORDER BY에 명시된 컬럼이 제일 먼저 읽는 테이블에 속해야 한다. (조인 시 드라이빙 테이블)
2) ORDER BY 순서대로 생성된 인덱스가 있어야 한다.
3) WHERE 절에 첫 번째로 읽는 테이블의 칼럼에 대한 조건이 있다면?
    그 조건과 ORDER BY는 같은 인덱스를 사용할 수 있어야 한다.
4) B-Tree 인덱스만 사용가능
5) 조인의 경우는 Nested-loop 방식의 조인만
    = 실행계획에 join buffer가 포함된 경우, 순서가 흐트러져 인덱스를 사용하지 못할 수 있으니 주의
```

2. 조인에서 드라이빙 테이블만 정렬 = using filesort
- 조인이 수행되면 레코드 건수와 레코드의 크기는 거의 배수로 불어난다.
    = 가능하다면 드라이빙 테이블만 정렬한 다음 조인을 수행하는 것이 효율적
    = 이게 그거
```
조인에서 첫 번째로 읽히는 테이블의 컬럼만으로 ORDER BY 절을 작성해야 한다.
```


3. 조인에서 조인 결과를 임시 테이블로 저장 후 정렬 = using temporary; using filesort(p. 299)
- 정렬해야 할 레코드 건수가 가장 많으므로, 가장 느린 정렬방법

**NOTE**
- ORDER BY or GROUP BY는 WHERE 조건을 만족하는 레코드를 LIMIT 건수 만큼만 가져와서 처리 불가
    = 먼저 조건을 만족하는 레코드를 모두 가져와 정렬을 수행하거나 그루핑 작업을 수행한다.
    = 그다음 LIMIT
    = WHERE 조건이 잘 튜닝되어도 ORDER BY, GROUP BY 때문에 쿼리가 느려질 수 있다.

- 결과 처리: streaming vs buffering 
    = ORDER BY, GROUP BY는 streaming 처리가 불가능
    = 조건에 일치하는 레코드 모두 가져온 후 정렬/그루핑해서 보내야 하기 때문
    = LIMIT이 별로 도움이 되지 않는 이유
    (네트워크로 전송되는 건수는 줄어드나, MySQL 서버가 처리하는 작업양은 차이X)
    (일반적으로 LIMIT 1은 한 건을 바로 보낼 수 있으므로 streaming 방식에서는 빠르다.)
    (인덱스를 사용한 정렬은 streaming, 나머지는 buffering)

### In short,
핵심은

1. 어떤 테이블이 먼저 드라이빙되어 조인되는지
2. 어떤 정렬방식으로 처리되는지 

따라서 인덱스를 사용한 정렬이 최선 but 안된다면 적어도 드라이빙 테이블만 정렬하도록 튜닝필요
