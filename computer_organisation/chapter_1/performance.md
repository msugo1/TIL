## 성능
- 무 자르듯 한 가지로 딱 표현할 수 없다.
    = 관점에 따라 다르게 표현될 수 있다.
    (무엇을 중시하는가?)

ex.
for individual users
- 응답시간이 성능의 척도
(response time determined by execution time)

for managers in data centres
- 처리량, or 대역폭(일정시간 동안 처리한 작업의 양)이 척도
(throughput, and bandwidth)

```
성능x = 1 / 실행시간x

성능x > 성능y

1 / 실행시간x > 1 / 실행시간y

실행시간y > 실행시간x
```

## 성능의 측정
1. 시간
- 컴퓨터 성능의 가장 기본적인 척도
- 같은 작업을 최단 시간에 실행하는 컴퓨터가 가장빠른 컴퓨터
- 실행시간: 프로그램을 처리하는데 걸린 시간을 초 단위로 표시한 것
    = 외에도 응답시간/경과시간/벽시계 시간 ...
    = 한 작업을 끝내는데 필요한 전체시간(디스크 접근, 메모리 접근, 입출력 작업, 운영체제 오버헤드 등을 모두 더한시간)
- 프로세서 하나가 여러 프로그램을 동시에 실행하는 경우, 특정 프로그램의 경과시간 최소화 보다는 `처리량 최적화`가 중요할 수 있다.
    = CPU 실행시간
    = 입출력에 걸린 시간 or 다른 프로그램을 실행하는데 걸린시간 포함X

**사용자 CPU 시간**
- 프로그램 자체에 소비된 CPU 시간

**시스템 CPU 시간**
- 프로그램 수행을 위해서 운영체제가 소비한 CPU 시간

**시스템 성능**
- 다른 부하가 없는 시스템에서의 경과시간으로 계산

**CPU 성능**
- 사용자 CPU 시간으로 계산한 것

2. 클럭
- 하드웨어가 기본함수를 얼마나 빨리 처리할 수 있는지와 관련된 성능척도
- 거의 모든 컴퓨터는, 하드웨어 이벤트가 발생하는 시점을 결정하는 클럭을 이용해 만들어진다.

**클럭 사이클**
- 클럭의 시간간격
- 클럭주기
    = 한 클럭 사이클에 걸리는 시간이나 클럭속도로 표시
- 클럭 속도

### CPU 성능과 성능인자
- 프로그램 CPU 실행시간 = `프로그램 CPU 클럭 사이클 수 * 클럭 사이클 시간`
                        = `프로그램 CPU 클럭 사이클 수 / 클럭 속도`
- 클럭 사이클의 길이를 줄이거나, 프로그램 실행에 필요한 클럭 사이클 수를 줄이면 성능개선
    = 그러나 둘 중 하나를 감소시키면 다른 하나가 증가하는 경우가 자주 발생;;

### 명령어 성능
- 컴파일러가 실행할 명령어 생성 and 컴퓨터는 이 명령어 실행
    = 즉, 실행시간은 프로그램의 명령어 수와도 관련이 있다.
    == 실행시간 = 실행 명령어 수 * 명령어의 평균 실행시간

- 프로그램 실행에 필요한 클럭 사이클 수?
    == CPU 클럭 사이클 수 = 명령어 수 * 명령어 당 평균 클럭 사이클 수 
    (CPI = clock cycles per instruction)
- 명령어마다 모두 실행시간이 다르므로, CPI는 프로그램이 실행한 모든 명령어에 대한 `평균값`을 사용

### 고전적인 CPU 성능식
```
CPU 시간 = 명령어 개수 * CPI * 클럭 사이클 시간
         = (명령어 개수 * CPI) / 클럭 속도

클럭 속도는 클럭 사이클 시간의 역수
```

```
프로그램 실행 시간
= 초/ 프로그램
= (명령어 수 / 프로그램) * (클럭 사이클 수 / 명령어) * (초 / 클럭 사이클)
```
- 컴퓨터 성능에 대한 완벽하고 믿을만한 척도는 `실행시간`밖에 없다.
- 명령어 개수를 줄이기 위해 명령어 집합을 바꾸었을 때, 클럭 속도가 느려지거나 CPI 커져서 성능이 오히려 나빠질 수 있다.
- CPU 실행시간은 `실제 프로그램을 실행`시켜서 얻을 수 있다.


**한 가지 인자**만으로 성능을 평가해서는 안된다.
- 두 컴퓨터를 비교할 때는 실행시간에 영향을 미치는 세 가지 인자를 모두 고려해야 한다.
ex. 클럭 속도가 같으면 다른 인자를 비교해서 성능평가
- CPI는 `명령어 배합`에 따라 달라지므로, 클럭속도가 같더라도 `명령어 개수`와 `CPI`는 반드시 비교해야 한다.

### 프로그램 성능의 이해
- 프로그램의 성능은 `알고리즘`, `언어`, `컴파일러`, `컴퓨터 구조`, `실제 하드웨어`의 영향을 받는다.

**알고리즘**
- affected by 명령어 개수, CPI
- 소스 프로그램이 수행할 명령어의 수 == 프로세서가 수행할 명령어의 수
- 알고리즘이 느린지 혹은 빠른지에 따라 CPI 영향

**프로그래밍 언어**
- affected by 명령어 개수, CPI
- 프로그래밍 언어로 표현된 문장 -> 프로세서 명령어로 변환
    = 명령어 개수에 영향
- 언어적인 특성도 CPI에 영향을 미친다.
    ex. 자바 등의 언어처럼 추상화가 많이 된 경우, 간접 호출(indirect call)을 필요로 하므로 CPI가 높은 명령어를 사용

**컴파일러**
- affected by 명령어 개수, CPI
- 소스 언어를 컴퓨터 명령어로 변환하는 역할
    = 얼마나 효율적인가에 따라 `명령어 개수`와 `CPI`가 영향을 받음
- 컴파일러의 기능이 복잡한 만큼 CPI에 복잡한 영향을 준다.

**명령어 집합구조**
- affected by 명령어 개수, 클럭속도, CPI
- 명령어 집합구조는 어떤 기능을 수행하기 위해 `필요한 명령어의 수`, `각 명령어 당 사이클의 수`, `프로세서의 클럭속도`에 영향을 미치므로, CPU 성능의 세 인자에 모두 영향을 끼친다.

### IPC
- Instructions per clock cycle (클럭 사이클 당 명령어 수)
    = CPI의 역수
