### 32비트 수치 피연산자

대부분 16비트로 충분하지만, 때에 따라 더 큰 상수가 필요한 경우도 있다.
**lui(load upper immediate)**
- 레지스터의 상위 16비트에 상수를 넣는다.
- 하위 16비트는 다른 명령으로 채운다.
32비트가 상수를 가지게 된다.
```
lui $s0, 61
ori $s0, $s0, 2304
```

NOTE)
vs addi
1) addi
- 16비트 상수 필드의 Most Significant Bit를 상위 16비트에 복사하여 32비트 상수를 만들어 연산에 사용한다.
(부호확장)
2) ori
- 상위 16비트를 0으로 채운후 연산

따라서, 어셈블러는 ori + lui를 조합하여 32비트 상수를 만든다.

- 컴파일러나 어셈블러는 큰 숫자를 직접 다룰 수 없다.
    = 몇 조각으로 나누어서 레지스터에서 재조립
- 수치 명령어의 상수, 적재/저장 명령어의 메모리 주소도 상수 필드 크기의 제약이 문제가 된다.
    = 어셈블러가 이 문제를 해결하도록 하려면, 큰 값을 만드는데 사용할 임시 레지스터를 제공해야 한다.
    = `$at`

### 분기와 점프 명령에서의 주소지정
jump
- 6비트 op코드 + 26비트 주소필드

조건부 분기명령
- 분기주소외에 2개의 피 연산자가 더 있다.
ex. bne $s0, $s1, Exit
(bit)6  5    5    16
- 16비트 필드 안에 모든 주소를 담야아 한다.
    = 현실적으로 불가능
- 대안
`PC = 레지스터 + 분기주소`
    = 프로그램 크기가 2^32까지 커지는 것을 허용
    = 조건부분기도 지원하여 분기주소의 크기제한 극복
    = 조건부 분기는 주로 순환문/if문에서 사용
    = 가까이 있는 명령어로 분기하는 경향이 있다.
    = Program Counter는 현 명령어의 주소를 가지고 있으므로 분기 주소를 더할 레지스터로 PC를 선택하면 현 위치에서 +- 2^15 워드 이내 떨어진 곳은 어디든지 분기할 수 있다.
    = `PC 상대주소 지정`
- 실제 MIPS 주소는 현재 명령어 대신 `다음 명령어 주소(PC + 4)`를 기준으로 한다.

vs to procedure
- 프로시저들은 가까이 붙어있어야 할 이유가 없다.
    = jal 명령은 다른 주소지정 방식을 사용한다.

분기할 거리를 바이트 단위 대신 `워드 단위`로 나타내면 더 먼거리까지 분기가능

### MIPS 주소지정 방식 요약
1. 수치 주소지정
- 피 연산자는 명령어 내에 있는 상수

2. 레지스터 주소지정
- 피연산자는 레지스터
- 바이트, 하프워드, 워드를 전송하는 적재와 저장명령이 각각 따로 있음

3. 베이스 또는 변위 주소지정
- 메모리 내용이 피연산자
- 메모리 주소: 레지스터 + 명령어 내의 상수

4. PC 상대주소지정
- 주소: PC 값과 명령어 내 상수의 합

5. 의사직접 주소지정
- 명령어 내의 26비트를 PC의 상위비트들과 연접하여 점프주소 구하기.
